A — Atomicity (원자성)

트랜잭션 내 작업은 전부 성공하거나 전부 실패. 계좌 이체 시 출금과 입금은 반드시 같이 성공해야 함.

C — Consistency (일관성)

트랜잭션 전후 데이터는 항상 일관된(무결성을 만족하는) 상태.

I — Isolation (격리성)

동시에 실행되는 여러 트랜잭션이 서로의 중간 상태를 볼 수 없음.

D — Durability (지속성)

커밋된 트랜잭션은 장애가 나도 영구 보존됨.

🌳 인덱스 B-Tree 원리 시험 필수
[50] / \ [25] [75] / \ / \ [10] [30] [60] [90] 🔍 id = 60 검색 → 50보다 큼 → 오른쪽 → 75보다 작음 → 왼쪽 → 60 발견! (3번만에)
인덱스가 있을 때
100만 건 → 약 20번 비교 (log₂ 1,000,000)
인덱스가 없을 때
100만 건 → 최대 100만 번 비교 (Full Scan)
⚠️ 단점
INSERT/UPDATE/DELETE 시 인덱스도 갱신 → 쓰기 성능 저하

Q2. 트랜잭션의 ACID 중 "격리성(Isolation)"이란 무엇인가?

▶ 정답 보기

Q3. 인덱스를 생성하면 SELECT가 빨라지는데, 어떤 작업의 성능이 저하되는가?

▶ 정답 보기
INSERT, UPDATE, DELETE — 데이터 변경 시 인덱스도 함께 갱신해야 하기 때문에 쓰기 성능이 저하됨.

② SQL 실무 (PostgreSQL)

⚡ SQL 실행 순서 시험 필수
✍️ 작성 순서
SELECT → FROM → WHERE → GROUP BY → HAVING → ORDER BY
⚙️ 실행 순서
FROM → WHERE → GROUP BY → HAVING → SELECT → ORDER BY
📋 테이블 생성 (DDL)
-- 사용자 테이블 생성
CREATE TABLE users (
    id         SERIAL PRIMARY KEY,           -- 자동 증가 PK
    name       VARCHAR(100) NOT NULL,        -- 이름 (필수)
    email      VARCHAR(200) UNIQUE NOT NULL, -- 이메일 (중복 불가)
    age        INTEGER CHECK (age >= 0),     -- 나이 (0 이상)
    is_active  BOOLEAN DEFAULT true,         -- 기본값 true
    created_at TIMESTAMP DEFAULT NOW()       -- 등록일
);

-- 주문 테이블 (외래키 포함)
CREATE TABLE orders (
    id       SERIAL PRIMARY KEY,
    user_id  INTEGER REFERENCES users(id) ON DELETE CASCADE,
    -- ON DELETE CASCADE: 부모(users) 삭제 시 자식(orders)도 자동 삭제
    product  VARCHAR(200) NOT NULL,
    amount   INTEGER NOT NULL,
    order_at TIMESTAMP DEFAULT NOW()
);
🔗 JOIN 시각화 시험 필수
users 테이블 orders 테이블 ┌────┬──────┐ ┌────┬─────────┬─────────┐ │ id │ name │ │ id │ user_id │ product │ ├────┼──────┤ ├────┼─────────┼─────────┤ │ 1 │ 홍길동│ │ 1 │ 1 │ 노트북 │ │ 2 │ 김철수│ │ 2 │ 1 │ 마우스 │ │ 3 │ 이영희│ │ 3 │ 2 │ 키보드 │ └────┴──────┘ └────┴─────────┴─────────┘ INNER JOIN 결과: LEFT JOIN 추가 결과: 홍길동 | 노트북 이영희 | NULL (주문없음) 홍길동 | 마우스 김철수 | 키보드
-- INNER JOIN: 주문이 있는 사용자만 (교집합)
SELECT u.name, o.product
FROM users u
INNER JOIN orders o ON u.id = o.user_id;

-- LEFT JOIN: 주문 없는 사용자도 포함 (왼쪽 테이블 전체)
SELECT u.name, COALESCE(o.product, '주문없음') AS product
FROM users u
LEFT JOIN orders o ON u.id = o.user_id;
🔍 집계 함수 & 서브쿼리
-- GROUP BY + HAVING (2명 이상인 나이대만 조회)
SELECT age, COUNT(*) AS 인원수
FROM users
GROUP BY age
HAVING COUNT(*) >= 2
ORDER BY age;

-- 서브쿼리: 평균 나이보다 많은 사용자 조회
-- (괄호 안 먼저 실행 → 평균값 반환 → 비교)
SELECT name, age
FROM users
WHERE age > (SELECT AVG(age) FROM users);

-- PostgreSQL 고유: RETURNING — INSERT 후 결과 즉시 반환
INSERT INTO users (name, email)
VALUES ('테스트', 'test@test.com')
RETURNING id, created_at;

-- UPSERT: 있으면 UPDATE, 없으면 INSERT
INSERT INTO users (email, name) VALUES ('hong@test.com', '홍길동')
ON CONFLICT (email) DO UPDATE SET name = EXCLUDED.name;
🧠 섹션 2 — 시험에 꼭 나오는 핵심 퀴즈

Q1. INNER JOIN과 LEFT JOIN의 차이를 설명하시오.

▶ 정답 보기
INNER JOIN: 양쪽 테이블 모두에 일치하는 행만 반환 (교집합). LEFT JOIN: 왼쪽 테이블의 모든 행을 반환하고, 오른쪽에 일치하는 행이 없으면 NULL로 채움.

Q2. 다음 SQL의 실행 순서를 나열하시오: SELECT name FROM users WHERE age > 20 GROUP BY name HAVING COUNT(*) > 1 ORDER BY name

▶ 정답 보기
FROM → WHERE → GROUP BY → HAVING → SELECT → ORDER BY

Q3. ON DELETE CASCADE와 ON DELETE RESTRICT의 차이는?

▶ 정답 보기
CASCADE: 부모 행 삭제 시 자식 행도 자동 삭제. RESTRICT: 자식 행이 존재하면 부모 행 삭제 자체를 막음 (오류 발생).

③ JPA 연동

🔄 JPA란? — Java 객체 ↔ DB 테이블 자동 매핑
개발자가 하는 일: Java 클래스(Entity) 작성 JPA가 하는 일: SQL 자동 생성 및 실행 users 테이블 User.java 클래스 ┌──────────────┐ ┌──────────────────┐ │ id (int) │ ←자동→ │ @Id Long id │ │ name (str) │ ←매핑→ │ String name │ │ email (str) │ ←매핑→ │ String email │ └──────────────┘ └──────────────────┘ 결과: SQL 없이 Java 코드로 DB 조작 가능!
🏷️ 핵심 어노테이션
@Entity                          // ① 이 클래스를 DB 테이블과 매핑
@Table(name = "users")           // ② 매핑할 테이블 이름 지정
public class User {

    @Id                          // ③ 이 필드가 기본키(PK)
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    // ④ PK 자동 생성 — PostgreSQL SERIAL에 위임
    private Long id;

    @Column(name = "name", nullable = false, length = 100)
    // ⑤ 컬럼명, null 허용 여부, 최대 길이 설정
    private String name;

    @Column(unique = true, nullable = false)
    private String email;

    @Column(name = "is_active", columnDefinition = "boolean default true")
    private Boolean isActive;

    protected User() {} // JPA는 기본 생성자 필수!

    public User(String name, String email) {
        this.name = name;
        this.email = email;
    }
}
어노테이션의미
@EntityJPA가 관리하는 엔티티 클래스
@Table매핑할 DB 테이블 이름 지정
@Id기본키(PK) 필드 지정
@GeneratedValue(IDENTITY)DB의 AUTO_INCREMENT / SERIAL에 PK 생성 위임
@Column컬럼명, nullable, length 등 세부 설정
@OneToMany1:N 관계 매핑
@ManyToOneN:1 관계 매핑 (FK가 있는 쪽)
💾 영속성 컨텍스트 & 변경 감지 시험 필수
[애플리케이션] [영속성 컨텍스트] [DB] │ │ │ │ find(User, 1) │ ← 캐시 없으면 │ │ ─────────────→ │ ─── SELECT ────→ │ │ │ ←── 결과 ────── │ │ ←────────────── │ (이후 1차 캐시 저장)│ │ │ │ │ find(User, 1) 재호출 │ │ ─────────────→ │ (캐시에 있음!) │ │ ←── 즉시 반환 ── │ (DB 쿼리 없음) │
// ✨ 변경 감지(Dirty Checking) — JPA의 마법
// UPDATE SQL을 직접 짜지 않아도 된다!

@Transactional
public void updateUserName(Long id, String newName) {
    User user = userRepository.findById(id).get();
    // 영속성 컨텍스트에서 관리 시작

    user.setName(newName);  // 단순 객체 값 변경
    // ↑ 트랜잭션 종료(커밋) 시점에 JPA가 변경을 감지하여
    //   자동으로 UPDATE users SET name = ? WHERE id = ? 실행!
    // 별도로 save()를 호출하지 않아도 됨
}
🗂️ Spring Data JPA Repository
// JpaRepository 상속만으로 CRUD 자동 완성!
public interface UserRepository extends JpaRepository<User, Long> {

    // 메서드명 규칙으로 쿼리 자동 생성
    List<User> findByName(String name);
    // → SELECT * FROM users WHERE name = ?

    List<User> findByAgeGreaterThanEqual(int age);
    // → SELECT * FROM users WHERE age >= ?

    Optional<User> findByEmail(String email);
    // → SELECT * FROM users WHERE email = ? LIMIT 1

    // @Query: 직접 JPQL 작성 (테이블명이 아닌 클래스명 사용!)
    @Query("SELECT u FROM User u WHERE u.age > :age AND u.isActive = true")
    List<User> findActiveUsersOlderThan(@Param("age") int age);
}

🧠 섹션 3 — 시험에 꼭 나오는 핵심 퀴즈

Q1. @GeneratedValue(strategy = GenerationType.IDENTITY)의 의미는?

▶ 정답 보기
PK 값 생성을 데이터베이스에 위임한다는 의미. PostgreSQL의 SERIAL처럼 DB가 자동으로 PK 값을 증가시켜 할당함.

Q2. JPA의 "변경 감지(Dirty Checking)"란 무엇인가?

▶ 정답 보기
영속성 컨텍스트가 관리하는 엔티티의 값이 변경되면, 트랜잭션 커밋 시점에 JPA가 자동으로 변경을 감지하여 UPDATE SQL을 실행하는 기능. 개발자가 별도 save()/update() 호출 불필요.

Q3. FetchType.LAZY와 FetchType.EAGER의 차이는?

▶ 정답 보기
LAZY(지연 로딩): 연관 엔티티를 실제 사용 시점에 SELECT. EAGER(즉시 로딩): 조회 시 연관 엔티티를 함께 즉시 로딩. 불필요한 쿼리를 줄이기 위해 기본적으로 LAZY 권장.

④ Docker 배포 실습

🐳 이미지 vs 컨테이너 — 붕어빵 비유
🏭
이미지 (Image)

= 붕어빵 틀

  • 실행 환경이 굳어진 "설계도"
  • 읽기 전용 (Read-Only)
  • Docker Hub에 공유 가능
  • 예: ubuntu:22.04
🐟
컨테이너 (Container)

= 붕어빵

  • 이미지를 실행한 인스턴스
  • 읽기/쓰기 가능
  • 같은 이미지로 여러 개 실행 가능
  • 예: ubuntu01, ubuntu02
🧱 Docker 레이어 구조 시험 필수
┌────────────────────────────────────┐ │ 컨테이너 레이어 (읽기/쓰기) │ ← docker run 시 생성 ├────────────────────────────────────┤ │ RUN apt install nginx 레이어 │ ← Dockerfile 레이어 3 ├────────────────────────────────────┤ │ RUN apt update 레이어 │ ← Dockerfile 레이어 2 ├────────────────────────────────────┤ │ FROM ubuntu:22.04 기반 레이어 │ ← Dockerfile 레이어 1 └────────────────────────────────────┘ ✅ 각 레이어는 캐시됨 → 변경된 레이어 이후만 재빌드 ✅ 이미지 레이어는 읽기 전용, 컨테이너 레이어만 쓰기 가능

🧠 섹션 4 — 시험에 꼭 나오는 핵심 퀴즈

Q1. docker run과 docker exec의 차이는?

▶ 정답 보기
docker run: 이미지로부터 새 컨테이너를 생성하고 실행. docker exec: 이미 실행 중인 컨테이너 내부에 명령어를 추가 실행 (컨테이너를 새로 만들지 않음).

Q2. Docker 이미지의 레이어(Layer) 구조의 장점은?

▶ 정답 보기
각 레이어가 캐시되므로, 이미지 재빌드 시 변경된 레이어 이후만 다시 빌드함. 이미지 다운로드 시에도 이미 있는 레이어는 재사용해 시간과 용량 절약.

Q3. docker run -p 5433:5432에서 앞의 5433과 뒤의 5432는 각각 무엇을 의미하는가?

▶ 정답 보기
앞(5433): Host(Windows) 포트. 뒤(5432): Container 내부 포트. localhost:5433으로 접속하면 컨테이너의 5432 포트로 전달됨. 포트 충돌 시 앞의 Host 포트를 다른 번호로 변경.
