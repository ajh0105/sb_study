📖 Chapter 04

⚙️ Service 계층

비즈니스 로직을 담당하는 Service 계층을 학습합니다. @Service, @Transactional, 의존성 주입까지 완전 정복!

⏱️ 예상 학습 시간: 45분 📝 핵심 어노테이션: 5개 🎯 퀴즈: 50문제

❓ 1. Service 계층이란?

Service는 비즈니스 로직(업무 처리)을 담당하는 계층입니다. Controller와 Repository 사이에서 실제 애플리케이션의 핵심 기능을 구현합니다.

Controller
요청/응답 처리
→
Service
비즈니스 로직
→
Repository
DB 접근
💡

왜 Service 계층을 분리할까?

  • 역할 분리: Controller는 HTTP 처리에만, Service는 비즈니스에만 집중
  • 재사용성: 여러 Controller에서 같은 Service 메서드 호출 가능
  • 테스트 용이: 비즈니스 로직만 독립적으로 테스트 가능
  • 트랜잭션 관리: Service에서 트랜잭션 경계 설정

🏷️ 2. 핵심 어노테이션

@Service

이 클래스가 Service 계층 컴포넌트임을 선언합니다. Spring이 자동으로 Bean으로 등록하여 의존성 주입에 사용합니다. @Component의 특화 버전입니다.

@Transactional

메서드 또는 클래스 전체를 하나의 트랜잭션으로 묶습니다. 도중에 오류 발생 시 자동 롤백됩니다. 클래스에 붙이면 모든 public 메서드에 적용됩니다.

@Transactional(readOnly = true)

읽기 전용 트랜잭션. 데이터를 조회만 할 때 사용합니다. JPA의 더티 체킹(변경 감지)을 비활성화하여 성능이 향상됩니다.

@RequiredArgsConstructor

Lombok이 final 필드를 인수로 받는 생성자를 자동 생성합니다. Spring의 생성자 주입에 활용됩니다.

🏖️ 3. TourService 전체 분석

TourService.java — 완전 분석
@Service                  // ① Spring Bean으로 등록
@RequiredArgsConstructor  // ② final 필드의 생성자 자동 생성 (의존성 주입)
@Transactional            // ③ 클래스 전체에 트랜잭션 적용
public class TourService {

    // final = 반드시 생성자에서 초기화 (불변성 보장)
    private final TourRepository tourRepository;
    private final ReviewRepository reviewRepository;
    private final FileConfig fileConfig;

    // ——— 관광지 등록 ———
    public Long save(TourFormDto dto) throws Exception {
        // 1. 파일 업로드 처리
        String fileName = FileUploadUtil.saveFile(fileConfig.uploadDir, dto.getFile());

        // 2. DTO → Entity 변환 (Builder 패턴)
        Tour tour = Tour.builder()
                .name(dto.getName())
                .category(dto.getCategory())
                .address(dto.getAddress())
                .tel(dto.getTel())
                .comment(dto.getComment())
                .thumb(fileName)
                .build();

        // 3. DB에 저장 후 ID 반환
        return tourRepository.save(tour).getId();
    }

    // ——— 관광지 수정 ———
    public void update(Long id, TourFormDto dto) throws Exception {
        // 1. ID로 기존 데이터 조회 (없으면 예외 발생)
        Tour tour = tourRepository.findById(id).orElseThrow();

        // 2. 새 파일이 있을 때만 이미지 교체
        if (dto.getFile() != null && !dto.getFile().isEmpty()) {
            String fileName = FileUploadUtil.saveFile(fileConfig.uploadDir, dto.getFile());
            tour.setThumb(fileName);
        }

        // 3. 필드 직접 수정 (Setter 사용)
        // ※ @Transactional 덕분에 save() 없이도 자동 저장됨 (더티 체킹)
        tour.setName(dto.getName());
        tour.setCategory(dto.getCategory());
        tour.setAddress(dto.getAddress());
    }

    // ——— 관광지 목록 ———
    public List<TourResponseDto> findAll() {
        // Stream + Lambda로 Entity → DTO 변환
        return tourRepository.findAll().stream().map(
                tour -> {
                    Double avg = reviewRepository.findAverageRating(tour.getId());
                    return new TourResponseDto(tour, RatingUtil.round(avg));
                }
        ).collect(Collectors.toList());
    }

    // ——— 관광지 삭제 ———
    public void delete(Long id) {
        tourRepository.deleteById(id);
    }
}
🔄

더티 체킹(Dirty Checking)이란?

JPA는 @Transactional 메서드 안에서 조회한 Entity의 필드 값이 변경되면, 트랜잭션 종료 시 자동으로 UPDATE 쿼리를 실행합니다.
update() 메서드에서 tourRepository.save(tour)를 호출하지 않아도 자동 저장되는 이유가 바로 더티 체킹입니다!

👤 4. MemberService 분석

MemberService.java
@Service
@RequiredArgsConstructor
@Transactional
public class MemberService {

    private final MemberRepository memberRepository;

    // 회원 가입
    public Long save(MemberFormDto dto) {
        Member member = Member.builder()
                .username(dto.getUsername())
                .password(dto.getPassword())  // ⚠️ tour 프로젝트: 평문 저장 (보안 취약)
                .name(dto.getName())
                .email(dto.getEmail())
                .role(MemberRole.USER)
                .build();
        return memberRepository.save(member).getId();
    }

    // 목록 조회 (읽기 전용 → 성능 최적화)
    @Transactional(readOnly = true)
    public List<Member> findAll() {
        return memberRepository.findAll();
    }

    // 아이디 중복 확인
    @Transactional(readOnly = true)
    public boolean existsByUsername(String username) {
        return memberRepository.existsByUsername(username);
    }

    // 로그인 처리
    @Transactional(readOnly = true)
    public Member login(String username, String password) {
        // 1. 아이디로 회원 조회 (없으면 예외)
        Member member = memberRepository.findByUsername(username)
                .orElseThrow(() -> new IllegalArgumentException("존재하지 않는 아이디입니다."));

        // 2. 비밀번호 확인 (⚠️ 평문 비교 — 실무에서는 BCrypt 사용!)
        if (!member.getPassword().equals(password)) {
            throw new IllegalArgumentException("비밀번호가 일치하지 않습니다.");
        }

        return member;
    }
}
🚨

보안 경고: 평문 비밀번호

tour 프로젝트는 비밀번호를 그대로(평문) 저장합니다. 실제 서비스에서는 절대 안 됩니다! Demo 프로젝트처럼 BCrypt로 암호화해야 합니다.
passwordEncoder.encode(dto.getPassword()) → BCrypt 해시로 저장
passwordEncoder.matches(입력값, 저장된해시) → 로그인 검증

💉 5. 의존성 주입 (Dependency Injection)

Service가 Repository를 사용하려면 Repository 객체가 필요합니다. 직접 new로 만들지 않고, Spring이 관리하는 객체를 주입받습니다.

3가지 의존성 주입 방법
// ❌ 방법1: 직접 생성 (Spring 관리 X, 권장 안 함)
private TourRepository tourRepository = new TourRepository();

// ⚠️ 방법2: @Autowired 필드 주입 (간단하지만 테스트 어려움)
@Autowired
private TourRepository tourRepository;

// ✅ 방법3: 생성자 주입 (권장! + Lombok @RequiredArgsConstructor 사용)
private final TourRepository tourRepository;

@RequiredArgsConstructor  // Lombok이 아래 생성자를 자동 생성
public class TourService {
    private final TourRepository tourRepository;
    // Lombok이 생성: public TourService(TourRepository tourRepository) { this.tourRepository = tourRepository; }
}

🌊 6. Stream과 Lambda 이해하기

findAll() 메서드에서 Entity 리스트를 DTO 리스트로 변환할 때 Stream과 Lambda를 사용합니다.

Stream + Lambda 단계별 이해
// ① tourRepository.findAll() → List<Tour> 반환
// ② .stream() → 리스트를 스트림(흐름)으로 변환
// ③ .map(tour → ...) → 각 Tour를 TourResponseDto로 변환
// ④ .collect(Collectors.toList()) → 다시 List로 수집

return tourRepository.findAll().stream()
    .map(tour -> {
        Double avg = reviewRepository.findAverageRating(tour.getId());
        return new TourResponseDto(tour, RatingUtil.round(avg));
    })
    .collect(Collectors.toList());

// for 루프로 동일한 코드 작성 시:
List<TourResponseDto> result = new ArrayList<>();
for (Tour tour : tourRepository.findAll()) {
    Double avg = reviewRepository.findAverageRating(tour.getId());
    result.add(new TourResponseDto(tour, RatingUtil.round(avg)));
}
return result;

✅ 7. 챕터 핵심 요약

  • @Service: Spring Bean 등록, Controller와 Repository 사이 계층
  • @Transactional: 트랜잭션 관리. 오류 시 자동 롤백
  • @Transactional(readOnly=true): 조회 전용. 더티 체킹 비활성화로 성능 향상
  • @RequiredArgsConstructor + private final: 생성자 주입 패턴
  • 더티 체킹: @Transactional 안에서 Entity 수정하면 save() 없이 자동 UPDATE
  • Builder 패턴: .builder().필드().build()로 Entity 생성
  • 실무에서는 비밀번호를 반드시 BCrypt로 암호화해야 함!
