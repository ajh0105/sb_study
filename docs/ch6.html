📦 DTO 패턴

데이터 전송 객체(DTO)가 왜 필요한지, Entity와 어떻게 다른지, Bean Validation으로 입력값을 검증하는 방법을 배웁니다.

⏱️ 예상 학습 시간: 40분 📝 핵심 어노테이션: 8개 🎯 퀴즈: 50문제

❓ 1. DTO란?

DTO (Data Transfer Object) = 데이터 전송 객체. 계층 간 데이터를 주고받기 위한 전용 클래스입니다.

🎯

왜 Entity를 직접 사용하지 않을까?

  • 보안: Entity에는 password, role 등 민감한 정보가 있음. DTO로 필요한 필드만 노출
  • 유연성: 화면마다 필요한 데이터가 다름. Entity를 그대로 쓰면 불필요한 데이터까지 노출
  • 검증: 입력 폼 데이터의 유효성 검사를 DTO에서 담당
  • 독립성: DB 구조(Entity)가 바뀌어도 DTO는 독립적으로 유지 가능

📝 3. Form DTO — 입력 데이터 수신

사용자가 폼에서 입력한 데이터를 받는 DTO입니다. Bean Validation 어노테이션으로 입력값을 검증합니다.

TourFormDto.java
@Getter @Setter
public class TourFormDto {

    private Long id;  // 수정 시 사용 (등록 시는 null)

    @NotBlank(message = "이름은 필수입니다.")   // 빈 문자열 불가
    private String name;

    @NotNull(message = "카테고리를 선택하세요.")  // null 불가
    private TourCategory category;

    private String address;
    private String tel;
    private String comment;

    // 파일 업로드용
    private MultipartFile file;  // org.springframework.web.multipart.MultipartFile

    private String trafficMap;
    private String tourMap;
    private String foodMap;
    private String bedMap;
}

📤 4. Response DTO — 응답 데이터 전달

Service에서 View(HTML)로 데이터를 전달할 때 사용하는 DTO입니다. Entity에서 필요한 필드만 추출하고, 계산된 값(평균 별점)도 포함합니다.

TourResponseDto.java
@Getter
public class TourResponseDto {

    private final Long id;
    private final String name;
    private final TourCategory category;
    private final String thumb;
    private final String address;
    private final String tel;
    private final Double avgRating;  // Entity에 없는 계산값!

    // Entity → DTO 변환 생성자
    public TourResponseDto(Tour tour, Double avgRating) {
        this.id = tour.getId();
        this.name = tour.getName();
        this.category = tour.getCategory();
        this.thumb = tour.getThumb();
        this.address = tour.getAddress();
        this.tel = tour.getTel();
        this.avgRating = avgRating;  // 계산된 평균 별점
    }
}
💡

DTO 이름 규칙

  • XXXFormDto: 폼 데이터를 받을 때 (Controller → Service 방향)
  • XXXResponseDto: 응답 데이터 (Service → Controller/View 방향)
  • LoginDto: 로그인 전용 DTO

✔️ 5. Bean Validation (입력값 검증)

MemberFormDto.java — 검증 어노테이션 예시
@Getter @Setter
public class MemberFormDto {

    @NotBlank(message = "아이디를 입력하세요.")
    @Size(min = 4, max = 20, message = "아이디는 4~20자입니다.")
    private String username;

    @NotBlank(message = "비밀번호를 입력하세요.")
    @Size(min = 6, message = "비밀번호는 6자 이상입니다.")
    private String password;

    @NotBlank(message = "이름을 입력하세요.")
    private String name;

    @Email(message = "이메일 형식이 아닙니다.")
    private String email;
}

// Controller에서 @Valid로 검증 활성화
@PostMapping("/new")
public String save(@Valid MemberFormDto dto,    // @Valid = 검증 실행
                    BindingResult bindingResult) { // 검증 결과 저장
    if (bindingResult.hasErrors()) {
        return "member/form";  // 검증 실패 → 폼으로 돌아가기
    }
    // 검증 통과 → 저장 처리
}
어노테이션의미예시
@NotNullnull 불가Enum 타입 등
@NotBlanknull, 빈 문자열, 공백 불가String 필드
@NotEmptynull, 빈 문자열 불가 (공백 허용)String 필드
@Size(min, max)문자열 길이 범위비밀번호, 아이디
@Email이메일 형식 확인이메일 필드
@Min(value)최솟값별점 >= 1
@Max(value)최댓값별점 <= 5
@Pattern(regexp)정규식 패턴 확인전화번호 형식

✅ 7. 챕터 핵심 요약

  • DTO = 계층 간 데이터를 주고받는 전용 클래스
  • Entity를 직접 쓰지 않는 이유: 보안, 유연성, 검증 분리
  • Form DTO: 사용자 입력 받기 + 검증
  • Response DTO: 뷰에 필요한 데이터만 추려서 전달
  • @Valid: DTO의 Bean Validation 어노테이션 활성화
  • BindingResult: 검증 결과 저장, hasErrors()로 오류 확인
  • 주요 검증: @NotBlank, @Size, @Email, @Min, @Max
