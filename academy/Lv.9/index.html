복습 핵심

  • Controller는 이제 "판단"이 아니라 "흐름 제어"에 집중해야 합니다.
이번 레벨 연결
[브라우저 요청] -> [Controller: 입력 바인딩 + 검증 + 라우팅] -> [Service: 비즈니스 규칙] -> [Controller: Model/Redirect 응답] Controller 품질 = 사용자 경험 품질 (에러 메시지, 재입력, 이동 흐름)

1) Controller 기초: 메서드 문법을 정확히 읽기

Deep Explaining

Controller는 "HTTP 요청"을 "서비스 호출"로 바꾸는 통역사입니다. 메서드 선언 한 줄에 이미 대부분의 의도가 들어 있습니다.

기초 코드 (라인별 주석) 
package com.example.lv9.controller; // 컨트롤러 패키지

import com.example.lv9.dto.MemberJoinRequest; // 입력 DTO
import com.example.lv9.service.MemberService; // 서비스 계층
import jakarta.validation.Valid; // DTO 검증 실행
import lombok.RequiredArgsConstructor; // final 필드 생성자 주입
import org.springframework.stereotype.Controller; // MVC 컨트롤러 선언
import org.springframework.ui.Model; // 뷰로 데이터 전달
import org.springframework.validation.BindingResult; // 검증 결과 객체
import org.springframework.web.bind.annotation.GetMapping; // GET 매핑
import org.springframework.web.bind.annotation.ModelAttribute; // 폼 바인딩
import org.springframework.web.bind.annotation.PostMapping; // POST 매핑
import org.springframework.web.bind.annotation.RequestMapping; // 공통 URL 매핑

@Controller // HTML 뷰를 반환하는 컨트롤러
@RequiredArgsConstructor // 생성자 주입
@RequestMapping("/members") // 공통 경로
public class MemberController {

    private final MemberService memberService; // 서비스 주입

    @GetMapping("/join") // 회원가입 폼 열기
    public String joinForm(Model model) { // public: 스프링 호출 가능, String: 뷰 이름 반환, Model: 뷰 데이터 상자
        model.addAttribute("req", new MemberJoinRequest()); // 폼 바인딩용 빈 DTO
        return "member/join"; // templates/member/join.html 렌더링
    }
}
Granular Analysis: public String joinForm(Model model)
부품문법적 의미왜 필요한가?
public접근 제어자DispatcherServlet(스프링)에서 호출 가능해야 함
String반환형어떤 뷰를 렌더링할지 이름으로 지정
joinForm메서드명메서드 역할(폼 진입점)을 드러냄
Model model파라미터컨트롤러 -> 템플릿 데이터 전달 통로

2) Controller 심화: 검증 실패/성공 흐름 설계

심화 코드 (라인별 주석) 
@PostMapping("/join") // 회원가입 제출 처리
public String join(
    @Valid @ModelAttribute("req") MemberJoinRequest req, // 폼 데이터를 req DTO로 바인딩 + 검증
    BindingResult br, // 검증 결과(에러 포함)
    Model model // 실패 시 화면에 데이터 재전달
) {
    // 1) 왜 BindingResult를 즉시 확인? -> 에러 상태에서 서비스 호출하면 잘못된 데이터가 전파됨
    if (br.hasErrors()) {
        model.addAttribute("pageTitle", "회원가입"); // 실패 화면 부가 데이터
        return "member/join"; // 입력 화면 재렌더링
    }

    // 2) 서비스는 규칙 담당. 컨트롤러는 호출/흐름만 담당
    Long memberId = memberService.join(req);

    // 3) 왜 redirect? -> 새로고침 시 중복 제출 방지(PRG 패턴)
    return "redirect:/members/detail?memberId=" + memberId;
}

@GetMapping("/detail")
public String detail(@org.springframework.web.bind.annotation.RequestParam Long memberId, Model model) {
    model.addAttribute("member", memberService.findById(memberId)); // 서비스 결과 DTO 전달
    return "member/detail";
}
Controller에서 자주 쓰는 파라미터 문법
문법역할주의점
@ModelAttribute폼 입력 바인딩폼 input name과 DTO 필드명이 맞아야 함
@ValidDTO 검증 실행검증 결과는 반드시 BindingResult로 확인
BindingResult검증 에러 보관@Valid 바로 다음 파라미터에 위치해야 안정적
@RequestParam쿼리 파라미터 수신필수/기본값 설정을 명확히 할 것
Model뷰 데이터 전달에러 재렌더링 시 필요한 데이터 누락 주의

3) Controller가 Service와 대화하는 방식 (복합 설명)

어떤 Service 메서드를 왜 호출하나?
  • memberService.join(req): 입력 DTO를 받아 가입 규칙(중복, 암호화)을 처리하기 위해 호출.
  • memberService.findById(memberId): 상세 화면에 필요한 응답 DTO를 가져오기 위해 호출.
  • Controller는 "무엇을 검증하고 어디로 보낼지"를 결정하고, 규칙 판단은 Service에게 위임합니다.
잘못된 구조 vs 올바른 구조 
// 잘못된 예: Controller가 비즈니스 판단까지 수행
// if (email already exists) ...

// 올바른 예: Controller는 입력/흐름, Service는 규칙
if (br.hasErrors()) return "member/join";
Long id = memberService.join(req);
return "redirect:/members/detail?memberId=" + id;
규칙이 Controller로 새면 테스트도 어려워지고, 코드 중복도 커집니다.

4) 어노테이션/키워드 사전 (Controller 편)

동작 원리 + 사용 시점
  • @Controller: HTML 뷰 반환 컨트롤러 선언.
  • @GetMapping/@PostMapping: HTTP 메서드별 라우팅.
  • @RequestMapping: 공통 URL prefix 지정.
  • @ModelAttribute: 폼 데이터 -> DTO 자동 바인딩.
  • @Valid + BindingResult: 검증 + 오류 처리 표준 패턴.
  • redirect:: POST 성공 후 중복 제출 방지 패턴.

5) Lv.9 실습 체크리스트

완료 기준
  1. public String joinForm(Model model)를 문법 단위로 설명할 수 있다.
  2. 검증 실패 흐름(BindingResult)과 성공 흐름(redirect)을 구분할 수 있다.
  3. Controller가 어떤 Service 메서드를 왜 호출하는지 설명할 수 있다.
  4. Controller와 Service의 책임 경계를 명확히 설명할 수 있다.
