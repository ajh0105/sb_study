0) 이전 레벨 복습 (Lv.2)

복습 핵심
  • Lv.2에서 Repository(JpaRepository 상속, Optional/List)를 배웠습니다.
  • 핵심은 "조회 결과 타입을 설계 의도에 맞게 고르는 것"이었습니다.
이번 레벨 연결
[Lv.2 Repository] 는 [Lv.3 Entity 설계 품질] 에 100% 의존합니다. 엔티티가 불안정하면 레포지토리 메서드 이름도 흔들리고, 서비스 규칙도 꼬입니다.

1) 환경 설정 복습 (Entity 학습 관점)

application.yml (왜 필요한지 라인 해설) 
spring: # 스프링 설정 최상위
  datasource: # DB 접속 정보
    url: jdbc:postgresql://localhost:5432/spring_lv1 # 어느 DB에 엔티티 테이블을 만들지 주소 지정
    username: postgres # DB 아이디
    password: 1234 # DB 비밀번호(학습용)

  jpa: # 엔티티 매핑 설정
    hibernate:
      ddl-auto: update # 엔티티 코드 변경을 테이블 구조에 반영
    show-sql: true # 엔티티 설계 결과로 어떤 SQL이 나가는지 확인
    properties:
      hibernate:
        format_sql: true # SQL 가독성 향상
왜 Entity 챕터에서 yml을 다시 보나?
  • ddl-auto가 어떻게 설정되느냐에 따라 엔티티 변경이 DB에 반영되는 방식이 달라집니다.
  • Entity를 학습할 때는 SQL 출력(show-sql)을 보며 "내 설계가 실제 쿼리로 어떻게 번역되는지" 확인해야 합니다.

2) 데이터 설계: Board 엔티티를 왜 이렇게 만들까?

필드 선정 사고방식
  • id: 게시글을 유일하게 식별하는 기본키.
  • title: 목록에서 바로 보여줄 핵심 정보.
  • content: 본문은 길 수 있으므로 TEXT 전략 고려.
  • writer: 게시글은 작성자(Member)와 반드시 연결되어야 함.
Board Entity (모든 라인 주석) 
package com.example.lv3.entity; // 엔티티 패키지 선언

import jakarta.persistence.Column; // 컬럼 옵션 지정
import jakarta.persistence.Entity; // 엔티티 선언
import jakarta.persistence.FetchType; // 연관관계 조회 전략 지정
import jakarta.persistence.GeneratedValue; // PK 생성 전략
import jakarta.persistence.GenerationType; // PK 전략 타입
import jakarta.persistence.Id; // 기본키 지정
import jakarta.persistence.JoinColumn; // FK 컬럼명 지정
import jakarta.persistence.ManyToOne; // 다대일 연관관계
import lombok.Builder; // 빌더 패턴
import lombok.Getter; // getter 자동 생성
import lombok.NoArgsConstructor; // 기본 생성자 자동 생성

@Entity // JPA가 Board를 테이블로 인식
@Getter // 필드 조회 메서드 생성
@NoArgsConstructor // JPA 프록시 생성 시 기본 생성자 필요
public class Board { // 게시글 엔티티

    @Id // 기본키
    @GeneratedValue(strategy = GenerationType.IDENTITY) // DB auto increment 사용
    private Long id; // 게시글 번호

    @Column(nullable = false, length = 100) // null 금지, 최대 100자
    private String title; // 제목

    @Column(nullable = false, columnDefinition = "TEXT") // null 금지, 긴 본문 저장용
    private String content; // 본문

    @ManyToOne(fetch = FetchType.LAZY) // 게시글 여러 개가 회원 한 명을 참조
    @JoinColumn(name = "member_id", nullable = false) // FK 컬럼명 member_id, null 금지
    private Member writer; // 작성자

    @Builder // Board.builder() 문법 지원
    public Board(String title, String content, Member writer) { // 생성자
        this.title = title; // 제목 대입
        this.content = content; // 내용 대입
        this.writer = writer; // 작성자 대입
    }

    public void update(String title, String content) { // 엔티티 상태 변경 메서드
        this.title = title; // 수정된 제목 반영
        this.content = content; // 수정된 내용 반영
    }
}
Java 기초 연결: private / Long / String / 참조타입
요소 뜻 왜 이렇게 선택?
private 외부 직접 접근 금지 Service를 거치지 않은 무분별한 변경 방지
Long 객체형 정수 PK 용도로 넉넉한 범위 + null 상태 표현 가능
String 문자열 타입 title/content는 텍스트 데이터
Member writer 객체 참조 단순 숫자 FK가 아니라 도메인 관계를 코드로 표현

3) 어노테이션 사전 (Lv.3 Entity 편)

동작 원리 + 사용 시점
  • @Entity: 클래스가 JPA 관리 대상임을 선언.
  • @Id, @GeneratedValue: PK 선언과 생성 전략 지정.
  • @Column: null/길이/타입 옵션으로 DB 제약을 코드에 반영.
  • @ManyToOne: Board -> Member 관계를 객체 지향적으로 표현.
  • @JoinColumn: FK 컬럼명을 명시해 DB 스키마 의도를 분명히 함.
  • @Getter, @NoArgsConstructor, @Builder: 반복 코드 감소 + 안정적 생성.

4) Builder 패턴 심층 분석 (Entity 생성 관점)

예시 코드 
Board board = Board.builder() // 빌더 시작
    .title("스프링 학습 기록") // 제목 필드 지정
    .content("오늘은 엔티티를 배웠다") // 내용 필드 지정
    .writer(member) // 작성자 엔티티 연결
    .build(); // 최종 Board 객체 생성
왜 빌더가 좋은가?
  • 가독성: 어떤 값이 어떤 필드로 들어가는지 한눈에 보입니다.
  • 안전성: 생성자 파라미터 순서 실수를 줄입니다.
  • 확장성: 나중에 필드가 늘어나도 코드 수정 부담이 줄어듭니다.

5) Granular Analysis 예시 (요청하신 메서드 분해 유지)

public String joinForm(Model model) 한 줄 분석
부품 문법적 정체 왜 필요한가?
public 접근 제어자 스프링 프레임워크가 외부에서 호출 가능해야 하기 때문
String 반환형 MVC에서 반환 문자열은 뷰 템플릿 이름
Model model 파라미터 컨트롤러 -> 뷰로 데이터를 전달하는 상자
이번 레벨의 메인 주제는 Entity이지만, 요청하신 Granular Analysis 원칙을 유지하기 위해 메서드 문법 분해를 고정 섹션으로 포함했습니다.

6) Lv.3 실습 체크리스트

완료 기준
  1. Board 엔티티 필드(id/title/content/writer) 선정 이유를 설명할 수 있다.
