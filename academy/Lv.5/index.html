0) 이전 레벨 복습 (Lv.4)

복습 핵심
  • Lv.4에서 Repository를 만들고 Optional/List 반환 의도를 학습했습니다.
  • 즉, DB에서 "어떻게 가져올지"는 준비되었습니다.
이번 레벨 연결
[Repository 결과(Entity)] -> [DTO로 가공] -> [화면/API에 안전하게 전달] DTO는 Entity를 외부로 바로 노출하지 않게 막아주는 보호막입니다.

1) DTO가 왜 필요한가? (Deep Explaining)

핵심 개념

DTO(Data Transfer Object)는 데이터 전달용 객체입니다. Entity를 그대로 화면/API에 넘기면 비밀번호 같은 민감 데이터가 섞일 수 있습니다. 그래서 "입력 DTO"와 "응답 DTO"를 분리해 사용합니다.

초보자 관점 요약
  • Entity = DB 중심 모델
  • DTO = 화면/API 중심 모델
  • 둘을 분리하면 보안, 유지보수, 테스트가 쉬워집니다.

2) 코드: Request DTO / Response DTO 라인별 해설

MemberJoinRequest (입력 DTO) 
package com.example.lv5.dto; // DTO 패키지 선언

import jakarta.validation.constraints.Email; // 이메일 형식 검증
import jakarta.validation.constraints.NotBlank; // 빈 문자열 검증
import jakarta.validation.constraints.Size; // 길이 검증
import lombok.Getter; // getter 자동 생성
import lombok.Setter; // setter 자동 생성 (폼 바인딩용)

@Getter // 필드 읽기 메서드 생성
@Setter // 필드 쓰기 메서드 생성 (폼 입력값 바인딩)
public class MemberJoinRequest { // 회원가입 입력 DTO

    @NotBlank(message = "이메일은 필수입니다.") // null/""/공백 금지
    @Email(message = "이메일 형식이 올바르지 않습니다.") // 이메일 규칙 검사
    private String email; // 사용자 입력 이메일

    @NotBlank(message = "비밀번호는 필수입니다.")
    @Size(min = 8, message = "비밀번호는 8자 이상이어야 합니다.")
    private String password; // 사용자 입력 비밀번호(평문)
}
MemberResponse (출력 DTO) 
package com.example.lv5.dto;

import com.example.lv5.entity.Member; // Entity -> DTO 변환에 필요

public record MemberResponse(Long id, String email) { // 응답은 필요한 필드만 노출

    public static MemberResponse from(Member member) { // 변환 팩토리 메서드
        return new MemberResponse(member.getId(), member.getEmail()); // password는 의도적으로 제외
    }
}

3) DTO <-> Entity 변환 기초

요청 DTO를 엔티티로 바꾸는 기본 패턴 
public Member toEntity(PasswordEncoder encoder) { // 입력 DTO -> Entity
    return Member.builder() // 빌더 시작
        .email(this.email) // DTO 필드 매핑
        .password(encoder.encode(this.password)) // 변환 시 암호화 강제
        .build(); // Entity 생성
}
왜 변환 메서드를 두나?
  • Service에서 변환 규칙을 반복 작성하지 않아도 됩니다.
  • 암호화/기본값 처리 같은 규칙을 한 곳에 모을 수 있습니다.
  • 코드 리뷰 시 "어디서 변환하는지"가 명확해집니다.

4) Java 기초 + 문법 분해 (Granular Analysis 유지)

public static MemberResponse from(Member member) 분해
부품문법적 의미왜 필요한가?
public접근 제어자서비스/컨트롤러 어디서든 호출 가능하게
static정적 메서드객체 생성 없이 MemberResponse.from(...)로 바로 사용
MemberResponse반환형엔티티가 아닌 응답 DTO만 반환해 경계 유지
from메서드명"어디에서 변환되는지" 의미를 이름으로 명확화
(Member member)파라미터변환 원본 엔티티 전달
private / Long / String 다시 연결
  • private: DTO/Entity 필드를 외부에서 직접 건드리지 못하게 보호.
  • Long: 식별자(id) 타입으로 넉넉한 범위 제공.
  • String: 이메일/비밀번호 같은 문자열 데이터 표현.

5) 어노테이션 사전 (DTO 편)

동작 원리 + 사용 시점
  • @NotBlank: 공백 입력 차단. 폼 입력 검증의 기본.
  • @Email: 이메일 형식 검증. 잘못된 주소 사전 차단.
  • @Size: 길이 검증. 비밀번호 최소 길이 정책 강제.
  • @Getter/@Setter: 폼 바인딩/읽기에 필요한 메서드 자동 생성.
DTO의 검증 어노테이션은 Controller의 @Valid와 함께 동작합니다. (다음 레벨에서 Service/Controller 연결 시 본격 적용)

6) Lv.5 실습 체크리스트

완료 기준
  1. 왜 Entity와 DTO를 분리해야 하는지 설명할 수 있다.
  2. 입력 DTO와 응답 DTO를 각각 언제 쓰는지 구분할 수 있다.
  3. from(), toEntity() 변환 패턴을 이해하고 직접 작성할 수 있다.
  4. 검증 어노테이션(@NotBlank, @Email, @Size) 역할을 설명할 수 있다.
