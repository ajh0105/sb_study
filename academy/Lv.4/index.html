1) 환경 설정 복습 (Repository 관점)

application.yml 필수 줄 (왜 필요한가?) 
spring:
  datasource:
    url: jdbc:postgresql://localhost:5432/spring_lv1 # Repository가 실제로 접근할 DB 주소
    username: postgres # DB 인증 계정
    password: 1234 # DB 인증 비밀번호

  jpa:
    show-sql: true # Repository 메서드가 어떤 SQL로 변환되는지 확인 가능
    properties:
      hibernate:
        format_sql: true # SQL 출력 가독성 향상
왜 Repository 챕터에서 SQL 로그가 중요할까?
  • 메서드 이름(findByEmailContaining)이 실제 어떤 SQL로 번역되는지 눈으로 확인할 수 있습니다.
  • 성능 문제(불필요 쿼리) 발견의 시작점이 SQL 로그입니다.

2) Repository 기본: JpaRepository 상속 라인별 해설

코드 (한 줄 주석형) 
package com.example.lv4.repository; // Repository 패키지 선언

import com.example.lv4.entity.Member; // 어떤 엔티티를 다룰지 지정
import org.springframework.data.jpa.repository.JpaRepository; // JPA 기본 CRUD 상속

public interface MemberRepository extends JpaRepository<Member, Long> { // Member 대상, PK 타입 Long
    // 이 줄 하나로 save/findById/findAll/deleteById 등 기본 메서드 사용 가능
}
Granular Analysis: interface 선언 한 줄 해부
코드 부품 문법적 의미 왜 필요한가?
public 접근 제어자 스프링이 컴포넌트 스캔 후 프록시 구현체를 외부에서 사용 가능하게
interface 규약 선언 타입 구현을 직접 쓰지 않고 스프링이 런타임에 자동 구현
extends 상속 검증된 CRUD 메서드를 재사용해 생산성/안정성 확보
JpaRepository<Member, Long> 제네릭 타입 지정 대상 엔티티와 PK 타입을 정확히 알려줘야 프레임워크가 올바르게 동작

3) Optional과 List 심화: 메서드 설계의 핵심

확장 메서드 선언 (라인별 이유) 
import java.util.List; // 여러 건 결과 반환 타입
import java.util.Optional; // 값이 없을 수도 있는 단건 결과 타입

public interface MemberRepository extends JpaRepository<Member, Long> {

    Optional<Member> findByEmail(String email); // email은 유니크 의도 -> 0 또는 1건

    List<Member> findByEmailContaining(String keyword); // 검색은 여러 건 가능 -> List

    List<Member> findByIdIn(List<Long> ids); // 전달받은 id 목록에 포함되는 회원들 조회

    boolean existsByEmail(String email); // 데이터 본문 없이 존재 여부만 확인
}
Optional vs List를 초등학생도 이해하게
  • Optional은 "상자 안에 1개가 있거나 없거나"입니다.
  • List는 "상자 안에 여러 개가 들어갈 수 있음"입니다.
  • 유니크 값 조회(email)는 Optional, 검색 결과(contains)는 List가 자연스럽습니다.
Service에서 Optional 사용 예 
Member member = memberRepository.findByEmail(email) // Optional 반환
    .orElseThrow(() -> new IllegalArgumentException("해당 이메일 회원이 없습니다.")); // 없으면 예외
Service에서 List 사용 예 
List<Member> members = memberRepository.findByEmailContaining(keyword); // 다건 조회
if (members.isEmpty()) {
    // 검색 결과가 없을 때 처리
}

4) Repository 메서드 네이밍 규칙 심화

왜 메서드 이름이 쿼리가 될까?

Spring Data JPA는 메서드 이름을 문법처럼 해석합니다. 예를 들어 findByEmailContaining는 "email 컬럼에서 keyword 포함" 조건으로 JPQL을 생성합니다.

자주 쓰는 패턴
메서드명 의미
findByEmailemail 일치 단건(의도상)
findByEmailContainingemail 부분 검색 다건
existsByEmail존재 여부 true/false
findByIdInid 목록 포함 다건
메서드명이 너무 길어지면 다음 레벨(커스텀 리포지토리/QueryDSL)에서 분리하는 것이 좋습니다.

5) 어노테이션/문법 사전 (Repository 편)

필수 키워드 정리
  • JpaRepository: CRUD + 페이징 + 정렬 기본 인터페이스.
  • Optional: null 실수를 줄이는 단건 안전 컨테이너.
  • List: 다건 조회 표준 타입.
  • boolean existsBy...: "있냐/없냐"만 빠르게 판단할 때 사용.

6) Lv.4 실습 체크리스트

완료 기준
