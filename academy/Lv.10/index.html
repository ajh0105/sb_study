2) 패키지별 핵심 역할 + 문법 공식

Entity 패키지 
// 문법 공식
@Entity
@Getter
@NoArgsConstructor
public class Xxx {
    @Id @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;

    @Column(nullable = false, length = ...)
    private String field;

    @Builder
    public Xxx(...) { ... }
}
  • 역할: DB 테이블 구조를 코드로 표현.
  • 핵심: 제약조건(@Column)을 엔티티에서 명확히 선언해 무결성 확보.
Repository 패키지 
// 문법 공식
public interface XxxRepository extends JpaRepository<Xxx, Long> {
    Optional<Xxx> findByUniqueField(...);
    List<Xxx> findBySearchFieldContaining(...);
    boolean existsByUniqueField(...);
}
  • 역할: DB 접근 창구.
  • 핵심: Optional/ List/ boolean 반환 타입을 의도에 맞게 선택.
DTO 패키지 
// 문법 공식
public class XxxRequestDto { ... } // 입력 모델
public record XxxResponseDto(...) {
    public static XxxResponseDto from(Entity e) { ... } // Entity -> DTO
}

public Entity toEntity(...) { ... } // DTO -> Entity
  • 역할: 계층 간 데이터 전달 경계.
  • 핵심: Entity를 외부에 직접 노출하지 않기.
Config 패키지 
// 문법 공식
@Configuration
public class XxxConfig {
    @Bean
    public SomeType someBean() { ... }
}
  • 역할: 공통 도구/설정 중앙 관리.
  • 핵심: Service/Controller에서 재사용할 Bean을 한 곳에 등록.
Service 패키지 
// 문법 공식
@Service
@RequiredArgsConstructor
@Transactional
public class XxxService {
    private final XxxRepository repo;

    public ReturnType useCase(RequestDto req) {
        // 규칙검증 -> 변환 -> 저장/조회 -> 변환 -> 반환
    }

    @Transactional(readOnly = true)
    public ResponseDto query(...) { ... }
}
  • 역할: 비즈니스 규칙의 중심.
  • 핵심: "Controller는 흐름, Service는 판단" 원칙 유지.
Controller 패키지 
// 문법 공식 (MVC)
@Controller
@RequestMapping("/prefix")
public class XxxController {
    @GetMapping("/form")
    public String form(Model model) { ... }

    @PostMapping("/save")
    public String save(@Valid @ModelAttribute("req") ReqDto req, BindingResult br) { ... }
}
  • 역할: HTTP 요청 수신 + 화면 라우팅.
  • 핵심: 검증 실패 처리와 redirect 흐름(PRG 패턴).

3) 실제 요청 1개를 끝까지 추적 (회원가입 시나리오)

코드 흐름 요약 
// Controller
@PostMapping("/members/join")
public String join(@Valid @ModelAttribute("req") MemberJoinRequest req, BindingResult br) {
    if (br.hasErrors()) return "member/join"; // 입력 오류 처리
    Long memberId = memberService.join(req); // Service 위임
    return "redirect:/members/detail?memberId=" + memberId;
}

// Service
@Transactional
public Long join(MemberJoinRequest req) {
    if (memberRepository.existsByEmail(req.getEmail())) throw new IllegalArgumentException("중복 이메일");
    Member saved = memberRepository.save(req.toEntity(passwordEncoder));
    return saved.getId();
}

// Repository
boolean existsByEmail(String email);
Member save(Member member);
왜 이 순서인가?
  1. Controller: 잘못된 입력을 먼저 걸러서 Service 부담 감소.
  2. Service: 도메인 규칙(중복/암호화/변환) 수행.
  3. Repository: 실제 DB 접근.
  4. Controller: 성공 시 새 요청으로 이동해 중복 제출 방지.

4) 어노테이션 총정리 (Lv.1~Lv.9)

사용된 주요 어노테이션 정리표
어노테이션주 사용 패키지역할핵심 사용 시점
@EntityEntity테이블 매핑도메인 모델 설계 시
@Id, @GeneratedValueEntity기본키/생성전략PK 필드 선언 시
@ColumnEntity컬럼 제약null/길이/유니크 제어
@ManyToOne, @JoinColumnEntity연관관계/FKBoard->Member 연결 시
@Getter, @SetterEntity/DTOLombok 메서드 생성반복 코드 감소
@NoArgsConstructor, @BuilderEntity기본생성자/빌더JPA 호환 + 가독성 향상
@Configuration, @BeanConfig공통 도구 등록PasswordEncoder 등 등록
@ServiceService비즈니스 로직 계층규칙 구현 시
@TransactionalService트랜잭션 경계쓰기/읽기 메서드 구분 시
@ControllerControllerMVC 컨트롤러화면 반환 시
@RequestMapping, @GetMapping, @PostMappingControllerURL 매핑HTTP 진입점 설계 시
@ModelAttribute, @RequestParamController요청값 바인딩폼/쿼리 파라미터 수신 시
@ValidController/DTO검증 실행입력 검증 시

5) 자주 틀리는 포인트 총정리

실수 -> 해결
  • 실수: Controller에서 중복 검사까지 처리 -> 해결: Service로 이동.
  • 실수: Entity를 그대로 응답 -> 해결: Response DTO 변환 고정.
  • 실수: 검증 실패인데 Service 호출 -> 해결: BindingResult 즉시 체크.
  • 실수: readOnly 없이 모든 메서드 @Transactional -> 해결: 조회/쓰기 분리.
