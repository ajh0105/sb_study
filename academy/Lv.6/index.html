1) DTO 심화 핵심 이론

Deep Explaining

DTO 심화의 핵심은 3가지입니다. (1) 상황별 검증 규칙 분리, (2) 중첩 객체 검증, (3) 매핑 일관성. 이 3개가 안 잡히면 프로젝트가 커질수록 입력 버그가 폭증합니다.

핵심 포인트
  • 회원 "생성"과 "수정"은 필요한 필드가 다르므로 검증 그룹을 분리합니다.
  • 중첩 DTO(예: 주소 객체)는 @Valid를 붙여야 내부 필드까지 검증됩니다.
  • 응답 DTO는 화면/API가 필요한 정보만 제공해 결합도를 낮춥니다.

2) 코드: DTO 심화 (라인별 주석)

검증 그룹 + 중첩 DTO 
package com.example.lv6.dto; // DTO 패키지

import jakarta.validation.Valid; // 중첩 객체 검증을 위해 필요
import jakarta.validation.constraints.Email; // 이메일 형식 검증
import jakarta.validation.constraints.NotBlank; // 공백 검증
import jakarta.validation.constraints.Size; // 길이 검증
import lombok.Getter;
import lombok.Setter;

public class ValidationGroups { // 검증 상황(생성/수정) 그룹 정의용 클래스
    public interface Create {} // 생성 시 규칙
    public interface Update {} // 수정 시 규칙
}

@Getter
@Setter
public class AddressDto { // 중첩 DTO 예시
    @NotBlank(message = "도시는 필수입니다.")
    private String city; // 도시

    @NotBlank(message = "상세 주소는 필수입니다.")
    private String detail; // 상세 주소
}

@Getter
@Setter
public class MemberRequestDto { // 생성/수정 공용 DTO

    @NotBlank(message = "이메일은 필수입니다.", groups = ValidationGroups.Create.class) // 생성 시 필수
    @Email(message = "이메일 형식이 아닙니다.", groups = {ValidationGroups.Create.class, ValidationGroups.Update.class})
    private String email; // 이메일

    @NotBlank(message = "비밀번호는 필수입니다.", groups = ValidationGroups.Create.class) // 생성 시만 필수
    @Size(min = 8, message = "비밀번호는 8자 이상", groups = ValidationGroups.Create.class)
    private String password; // 비밀번호

    @Valid // AddressDto 내부 필드(city/detail)까지 검증 실행
    private AddressDto address; // 중첩 DTO
}
응답 DTO + 매핑 메서드 
package com.example.lv6.dto;

import com.example.lv6.entity.Member;

public record MemberDetailResponse(
    Long id,
    String email,
    String city,
    String detail
) {
    public static MemberDetailResponse from(Member member) { // Entity -> DTO 변환
        return new MemberDetailResponse(
            member.getId(),
            member.getEmail(),
            member.getCity(), // 예시: 엔티티에 city/detail 필드가 있다고 가정
            member.getDetail()
        );
    }
}

3) Granular Analysis: 메서드 한 줄 분해

public static MemberDetailResponse from(Member member)
부품문법적 의미왜 필요한가?
public접근 제어자서비스/컨트롤러 어디서든 호출 가능하게
static정적 메서드객체 생성 없이 변환 가능
MemberDetailResponse반환형응답 모델을 고정해 API 계약 안정화
from메서드명"무엇에서 무엇으로" 변환하는지 명확히 표현
(Member member)파라미터변환 원본 엔티티 전달

4) DTO 심화 실수 포인트와 해결

자주 하는 실수
  • 실수 1: 생성/수정 검증 규칙을 같은 DTO에 무작정 섞음 -> 그룹 검증으로 분리
  • 실수 2: 중첩 DTO에 @Valid 누락 -> 내부 필드 검증이 동작하지 않음
  • 실수 3: Entity를 그대로 응답 -> 보안/결합도 문제
해결 전략
  • 규칙이 다르면 DTO를 분리하거나 Validation Group을 사용
  • 중첩 객체는 항상 @Valid 확인
  • 응답은 Response DTO + from() 매핑 고정

5) 어노테이션 사전 (DTO 심화 편)

동작 원리 + 사용 시점
  • @Valid: 중첩 객체 검증까지 확장.
  • groups = ...: 같은 DTO에서도 상황별 검증 규칙 분리.
  • @NotBlank, @Email, @Size: 입력 품질 보장 기본 세트.
  • record: 불변 응답 DTO 작성에 유리.

6) Lv.6 실습 체크리스트

완료 기준
  1. 검증 그룹(Create/Update) 개념을 설명할 수 있다.
  2. 중첩 DTO 검증에서 @Valid가 왜 필요한지 설명할 수 있다.
  3. from() 매핑 메서드로 응답 구조를 통일할 수 있다.
  4. DTO 심화 규칙을 팀 코딩 규칙으로 문서화할 수 있다.
