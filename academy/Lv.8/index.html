복습 핵심

이번 레벨 연결
[Controller 요청 DTO] -> [Service: 규칙 판단/검증/변환/트랜잭션] -> [Repository: DB 질의/저장] -> [Service: Entity를 DTO로 가공] -> [Controller 응답]

1) Service가 왜 어려운가? 그리고 왜 중요한가?

Deep Explaining

Service는 단순 저장 계층이 아닙니다. "규칙을 적용하는 판단 계층"입니다. 무엇을 저장할지, 무엇을 거절할지, 어떤 예외를 던질지, 어떤 형식으로 반환할지 모두 Service에서 결정합니다. 그래서 초보자가 가장 어려워합니다.

Service의 5가지 책임
  1. 검증 보강: DTO 검증(@Valid) 이후 도메인 규칙 검증(예: 중복 이메일)
  2. 변환: DTO -> Entity, Entity -> DTO
  3. 레포지토리 오케스트레이션: 어떤 Repository 메서드를 어떤 순서로 호출할지 결정
  4. 트랜잭션 경계: 성공/실패 시 데이터 일관성 보장
  5. 예외 메시지 표준화: 사용자/개발자 관점 오류 전달

2) Service 구현 코드 (라인별 + 레포지토리 연결 이유)

MemberService 예시 
package com.example.lv8.service; // 서비스 패키지

import com.example.lv8.dto.MemberJoinRequest; // 입력 DTO
import com.example.lv8.dto.MemberResponse; // 출력 DTO
import com.example.lv8.entity.Member; // 도메인 엔티티
import com.example.lv8.repository.MemberRepository; // 회원 저장소
import lombok.RequiredArgsConstructor; // final 필드 생성자 자동 주입
import org.springframework.security.crypto.password.PasswordEncoder; // 비밀번호 암호화 도구
import org.springframework.stereotype.Service; // 서비스 계층 선언
import org.springframework.transaction.annotation.Transactional; // 트랜잭션 경계

import java.util.List; // 다건 응답용

@Service // 스프링이 서비스 빈으로 관리
@RequiredArgsConstructor // final 필드(memberRepository, passwordEncoder) 생성자 주입
@Transactional // 기본적으로 쓰기 트랜잭션. 예외 발생 시 롤백
public class MemberService {

    private final MemberRepository memberRepository; // DB 접근 창구
    private final PasswordEncoder passwordEncoder; // Config에서 등록한 Bean 주입

    public Long join(MemberJoinRequest req) { // 회원가입 유즈케이스

        // 1) 왜 existsByEmail? -> 전체 회원을 가져오지 않고 존재 여부만 확인해 효율적
        if (memberRepository.existsByEmail(req.getEmail())) {
            throw new IllegalArgumentException("이미 사용중인 이메일입니다.");
        }

        // 2) 왜 DTO -> Entity 변환? -> DB 모델과 요청 모델 분리
        Member member = req.toEntity(passwordEncoder); // 내부에서 password encode 수행

        // 3) 왜 save 반환값을 받나? -> 저장 후 생성된 id 사용 가능
        Member saved = memberRepository.save(member);

        return saved.getId(); // 다음 화면 이동/응답에 필요한 식별자 반환
    }

    @Transactional(readOnly = true) // 조회 전용: 변경 감지 비용 감소
    public MemberResponse findOneByEmail(String email) {

        // 4) 왜 Optional + orElseThrow? -> null 분기 실수 방지 + 명확한 실패 흐름
        Member member = memberRepository.findByEmail(email)
            .orElseThrow(() -> new IllegalArgumentException("회원을 찾을 수 없습니다."));

        // 5) 왜 Entity를 직접 반환하지 않나? -> 민감정보 노출 방지
        return MemberResponse.from(member);
    }

    @Transactional(readOnly = true)
    public List<MemberResponse> searchByKeyword(String keyword) {

        // 6) 왜 List 조회 메서드? -> 검색은 다건 결과가 자연스러움
        List<Member> members = memberRepository.findByEmailContaining(keyword);

        // 7) 왜 map 변환? -> Entity 리스트를 응답 DTO 리스트로 변환
        return members.stream()
            .map(MemberResponse::from)
            .toList();
    }
}
레포지토리 메서드를 왜 이렇게 고르는가?
Service 요구사항사용 Repository 메서드선택 이유
이메일 중복 검사existsByEmail존재 여부만 필요하므로 가장 가볍고 빠름
회원 단건 조회findByEmail (Optional)없을 수 있으므로 Optional이 안전
회원 검색(다건)findByEmailContaining (List)검색 결과는 여러 개 가능
신규 회원 저장save엔티티 영속화 후 생성 ID 확보 가능

3) Service 문법 해부 (Granular Analysis)

public Long join(MemberJoinRequest req) 분해
부품문법적 의미왜 필요한가?
public접근 제어자Controller가 Service 메서드를 호출할 수 있어야 함
Long반환형저장 후 생성 ID를 다음 흐름(redirect/응답)에 사용
join메서드명유즈케이스(회원가입) 의도를 이름으로 명확히 표현
(MemberJoinRequest req)파라미터Controller가 전달한 입력 DTO를 받음
@Transactional vs @Transactional(readOnly=true)
  • @Transactional: 저장/수정/삭제 같은 쓰기 작업에 사용.
  • @Transactional(readOnly = true): 조회 메서드에서 의도 명확화 + 성능 최적화.
  • 원칙: 쓰기와 읽기를 메서드 단위로 분리해 명확하게 선언.

4) Service 실전 사고법 (초보자용 체크 포인트)

코드 작성 전 4문장 질문
  1. 이 메서드의 유즈케이스 이름은 무엇인가?
  2. 어떤 Repository 메서드를 어떤 순서로 호출해야 하는가?
  3. DTO/Entity 변환은 어디서 할 것인가?
  4. 성공/실패 시 어떤 값을 반환하거나 예외를 던질 것인가?
Service는 "코드 작성"보다 "순서 설계"가 먼저입니다. 순서가 맞으면 코드가 짧아져도 품질이 높습니다.

5) 어노테이션/키워드 사전 (Service 편)

동작 원리 + 사용 시점
  • @Service: 비즈니스 로직 계층 선언.
  • @RequiredArgsConstructor: final 의존성 생성자 주입 자동화.
  • @Transactional: 데이터 일관성을 보장하는 트랜잭션 경계.
  • Optional.orElseThrow: 값 없음 처리의 안전한 표준 패턴.
  • stream().map(...).toList(): Entity 리스트를 DTO 리스트로 변환.

6) Lv.8 실습 체크리스트

완료 기준
  1. Service가 왜 가장 중요한 계층인지 설명할 수 있다.
  2. Service 메서드마다 어떤 Repository 메서드를 왜 쓰는지 말할 수 있다.
  3. @Transactional과 readOnly 차이를 설명할 수 있다.
  4. DTO/Entity 변환 위치를 팀 규칙으로 고정할 수 있다.
